# ChronoOS Verification Matrix

Date: 2026-06-13

This is the compact evidence lookup for ChronoOS v0.1. It summarizes what has
actual QEMU or hardware evidence, what is only implemented in code, what is
blocked, and what remains roadmap/design-only.

Use this together with `docs/CURRENT_STATUS.md`, `docs/AI_PROGRESS_LOG.md`, and
`docs/release-checklist.md`. Do not upgrade a row unless the repo records the
actual command, log, screenshot, or hardware evidence.

## Verification Levels

- verified in QEMU
- verified in QEMU UEFI
- partially verified in QEMU
- partially verified in QEMU UEFI
- verified on hardware
- build fixed, boot not verified
- implemented in code, not verified
- blocked: OVMF unavailable
- blocked: QEMU unavailable
- blocked: build failure
- blocked by tooling
- blocked by environment
- broken
- needs manual verification
- roadmap/design-only

No current feature is `verified on hardware`.

## Best Next Verification Focus

The safest next engineering target is not a new feature. It is a narrow
build-compatibility or verification step:

1. Retest the window lifecycle gap for visual close confirmation, `tasks`, and
   `kill <observed-task-id>` on a fresh disposable QEMU run.
2. Retest the product-command input gap for `poster system` and
   `capsule current` with reliable manual input or a better input harness.
3. Run a controlled ChronoFS repair/recovery pass on throwaway images only,
   without touching the repo data disk.
4. Retest the userspace input/tooling gaps for `userspace elf`, `syshello`, and
   controlled `exec hello.elf` after `ld.lld` or equivalent safe tooling is
   available.
5. Investigate the UEFI loader `Out of Resources` failure next, then move to
   custom BIOS, SMP/AP, or ARP/UDP verification one narrow path at a time.
